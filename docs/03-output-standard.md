# قرارداد خروجی

## بسته استاندارد

| صفحه | نقش |
|---|---|
| E0 | یک سیستم در محیط و تعامل مستقیم با بازیگران/سرویس‌ها |
| P1 | معماری کلان و ستون فقرات جریان |
| R1 | خلاصه‌ی فرایند و توجیه معماری در یک صفحه |
| P2-A…N | یک صفحه برای بازکردن هر واحد P1 |
| P3 | blueprint دقیق، متصل و یک‌صفحه‌ای با دفتر تصمیم |

نمونه‌ی SnappStore هشت صفحه دارد: E0، P1، R1، P2-A، P2-B، P2-C، P2-D و P3.

## قواعد صفحه‌ها

### E0

فقط یک سیستم مرکزی و تعامل مستقیم با محیط نمایش داده شود. اجزای داخلی، مدل و پایگاه داده وارد E0 نشوند. ارتباط احتمالی خط‌چین باشد.

### P1

سیستم را در بالاترین سطح مفید به A، B، C، D و مانند آن بشکند. فناوری به صورت ریز زیر مسئولیت بیاید. نتیجه‌ی کسب‌وکاری و بازخورد فراموش نشود.

### R1

یک صفحه، عنوان اعتمادساز، فرایند قابل فهم و دلیل مرزهای مهم. فاقد chain-of-thought خصوصی.

### P2

هر صفحه دقیقاً یک واحد P1 را باز کند. عناصر داخل focus boundary ضخیم باشند و ورودی/خروجی والد بیرون بمانند. جزئیات جدید مجاز است؛ تغییر معنا یا تضاد ممنوع.

### P3

از یک صفحه فراتر نرود. اتصال‌ها فقط به اندازه‌ی روشن‌شدن وابستگی‌ها باشند. سه بخش تأییدشده، پیش‌فرض و باز را ثبت کند.

## تبار

```text
P1:A -> P2-A:A1..An
P1:B -> P2-B:B1..Bn
P1:C -> P2-C:C1..Cn
P1:D -> P2-D:D1..Dn
All P2 -> P3
```

قرارداد فرزندان باید قرارداد والد را کامل کند. تغییر P1 همه‌ی P2ها و P3 را نیازمند بازبینی می‌کند؛ تغییر P2 نیز P3 و رابط‌های آن را.

## زبان خط

- پیوسته: جریان اصلی یا تأییدشده.
- خط‌چین: اختیاری، کنترلی یا بالقوه.
- `?`: تصمیم باز.
- کادر ضخیم: مرز سیستم/زیرسیستم.
- مداد: یادداشت، فناوری، فرض و زمینه.
- ماژیک: معماری و قرارداد اصلی.
